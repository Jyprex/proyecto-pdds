## 1.1 Convención TO-BE vs AS-IS

1. **TO-BE (Diseño objetivo):** modelo de datos mínimo necesario para cumplir el comportamiento del sistema.
1. **AS-IS (Implementación actual) y mapeo:** entidades/estructuras presentes en el código actual.

## 2. Estructura de Datos (Arquitectura Híbrida)
El diseño de datos del sistema está compuesto por dos grandes pilares operando conjuntamente. Una base de datos estructurada persistente (motor SQL) y una arquitectura analítica en Memoria Principal (Memoria Heap de Java) que garantiza la latencia casi nula al procesar simulaciones inmensas bajo HGA/ALNS.

## 2.0 Diseño Objetivo (TO-BE) — Modelo mínimo

El diseño objetivo se limita a lo estrictamente necesario:

1. **Persistencia mínima:** Aeropuertos, Vuelos y Envíos (demanda).
1. **Estado transitorio de simulación:** sesión en memoria para streaming y consulta de progreso.
1. **DTOs de red:** snapshots espaciales y KPIs.

### 2.1 Modelo de Datos Transaccional y Auditoría (SQL)
Las tablas subyacentes mantienen entidades base estáticas y las bitácoras operacionales:

| Tabla / Entidad JPA | Clave Primaria | Responsabilidad Crítica |
|---|---|---|
| `aeropuertos` | `id` (Long) | Define capacidades, ICAO y metadatos geográficos. Control de concurrencia optimista (`@Version`) si aplica. |
| `vuelos` | `id` (Long) | Tramos operables (origen/destino, capacidad, horarios, cancelación). Alimenta el grafo en memoria del planificador. |
| `envios` | `id` (Long) | Demanda de maletas por fecha (origen/destino, cantidad). Fuente para agrupar `SuperLot` y ejecutar simulación multi-día. |

### 2.2 Base de Datos "En Caliente" (Estructuras de Memoria de Simulación)
Las simulaciones logísticas son tan veloces que saturarían el I/O del disco rígido. Por esto, los DTOs intermedios viven en la sesión temporal: `SimulationSessionState`.

| Propiedad `SimulationSessionState` | Tipo de Estructura | Finalidad |
|---|---|---|
| `sessionId` | String UUID | Llave unívoca del ConcurrentHashMap para separar clientes multitenant. |
| `status` | Enum (`IDLE`, `RUNNING`, `DONE`, `FAILED`) | **TO-BE:** estado formal de sesión consumible por UI. En implementación actual del simulador no existe `IDLE` en backend; el idle es estado de UI. |
| `airportLoads` | `Map<String, Integer>` | Carga actual de los N almacenes. Este mapa se recalcula cada 500ms simulados. Es el responsable de "colorear" a rojo un aeropuerto en React. |
| `activeRoutes` | `List<Map<String, Object>>` | Colección serializable de todos los aviones *en tránsito aéreo* y sus posiciones inter-nodales calculadas por vectores de tiempo. |
| `eventLog` | `List<String>` | Array histórico temporal del motor de eventos (Ruin, Repair, Salidas, Llegadas). |
| `reports` | `List<SimulationDayReport>` | Objeto resumen agrupado por días cronológicos (Ta) con metas como SLAs cumplidos y niveles de colapso. |
| `comparisonResults` | Object | Sólo se puebla al completar (`DONE`). Tiene data consolidada estricta (ALNS vs HGA) exportable. |

## 2.5 Implementación Actual (AS-IS) y Mapeo

1. Persistencia actual en el prototipo: entidades JPA `Aeropuerto`, `Vuelo`, `Envio`.
1. Estado en memoria de simulación: `SimulationProgressHolder.SimulationSessionState`.
1. Estado real de simulación: `SimulationProgressHolder.Status = RUNNING/DONE/FAILED`.
1. Estado de experimentación/exportación: `ExperimentSession.Status` y `ExperimentExportSession.Status` sí incluyen `IDLE`.

### 2.3 Objetos de Transferencia de Datos (DTOs) — Endpoint SSE
Dado el paradigma Server-Sent Events, el controlador emite tres streams paralelos con DTOs especializados para reducir al máximo el payload de transferencia.

#### `SimulationMapSnapshotDTO` (Proyectado en evento: "snapshot")
Encargado meramente de lo "Espacial y Geográfico".
* `status` (String)
* `simulatedTime` (String) | `currentEpochTime` (Long) -> Sincronización del Reloj Simulador.
* `activeRoutes` -> Lista acotada (ej. límite 220) para prevenir ahogo del navegador web del cliente. Posee variables como:
`from`, `to` (String ICAO)
`progress` (Double 0.0 - 1.0) para interpolación de posición en el mapa SVG (react-simple-maps).

#### `SimulationKpiSnapshotDTO` (Proyectado en evento: "kpi")
Encargado meramente de la "Salud Analítica y Topológica".
* `slaPercent` (Double) -> Porcentaje de demanda cumplida a tiempo.
* `globalOccupancy` (Double) -> Promedio ponderado.
* `airportLoads` (Map) -> Desagregado vital para colorear individualmente nodos.
* `criticalNodes` (Integer) -> Contador de aeropuertos >90% llenos.
* `isCollapseMode` (Boolean) -> Si corre el modelo catastrófico.

### 2.4 Decisiones de Diseño de Datos
* **Aislamiento Snapshot vs KPI:** Separar la geometría de rutas de los cálculos estadísticos en SSE disminuye latencias de serialización JSON. El Snapshot puede ser gigantesco (500 aviones) pero trivial, mientras el KPI es mínimo (10 variables) pero computacionalmente pesado.
* **Agregaciones Diferidas:** Las métricas de `totalBagsWaiting` o totales de SLAs de *días anteriores* no se reclasifican por cada paquete a nivel base de datos; la estructura `SimulationDayReport` funge como una tabla puente pre-agregada en memoria.
