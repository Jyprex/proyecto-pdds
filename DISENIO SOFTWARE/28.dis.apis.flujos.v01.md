## 2. Diseño de APIs y Flujos de Datos
El núcleo de integración entre el panel de monitoreo interactivo (React) y los motores algorítmicos en Java es la API Web. Debido a los volúmenes masivos de iteraciones que ocurren durante un experimento heurístico, los flujos se han dividido en operaciones HTTP tradicionales (Request-Response) para disparadores, y canales unidireccionales de alto rendimiento (Server-Sent Events) para la telemetría continua.

### 2.1 Catálogo de Endpoints REST Clásicos
Endpoints sincrónicos consumidos convencionalmente utilizando `fetch()`.

#### Módulo: Simulador v1 (`/api/v1/simulation`)
| Método | URL | Código HTTP | Descripción |
|---|---|---|---|
| **POST** | `/run/{dias}?algorithm=HGA&startDate=` | `202 Accepted` | Inicia una simulación paramétrica *Asíncrona*. El servidor no espera a que acabe; despacha el hilo al motor y devuelve inmediatamente un `sessionId`. |
| **POST** | `/run-collapse/{dias}` | `202 Accepted` | Mismo efecto que `/run`, pero inyecta factores de disrupción críticos forzando al algoritmo a colapsar para testear las capacidades de ruteo resiliente. |
| **GET** | `/status/{sessionId}` | `200 OK` | Endpoint de estado para polling. Retorna el `SimulationStatusDTO` con progreso, métricas y reportes acumulados. En el flujo UI actual se usa típicamente al cierre (`DONE`) para recolectar data pesada, pero puede consultarse en cualquier momento. |
| **POST** | `/export-excel/{sessionId}` | `200 OK` | Devuelve un binario comprimido `Blob` de tipo `application/vnd.openxmlformats...` generado dinámicamente con tablas de los días simulados. |
| **GET** | `/export/{sessionId}` | `200 OK` | Exportación CSV legacy de resultados por día. Retorna `409 Conflict` si la simulación aún está en curso. |

#### Módulo: Experimentos Numéricos DoE (`/api/v1/numeric-experiment`)
| Método | URL | Código HTTP | Descripción |
|---|---|---|---|
| **GET** | `/doe` | `200 OK` | Consulta el catálogo disponible del Diseño de Experimentos y configuraciones base para algoritmos. |
| **POST** | `/start` | `202 Accepted` | Lanza ejecuciones masivas (Batches) de iteraciones algorítmicas sin trazado gráfico. |
| **GET** | `/status/{sessionId}` | `200 OK` | Consulta el progreso/resultados parciales de una sesión DoE. |
| **POST** | `/export/start` | `202 Accepted` | Inicia la suite de exportación Excel (10 iteraciones × 5 niveles) y devuelve un `exportSessionId` para tracking. |
| **GET** | `/export/status/{exportSessionId}` | `200 OK` | Consulta el progreso de la exportación Excel. |
| **GET** | `/export/download/{exportSessionId}` | `200 OK` | Permite descargar el Excel generado cuando el estado es `DONE`. |

### 2.2 Flujo Continuo: Server-Sent Events (SSE)
Para mitigar los picos de congestión HTTP provocados por peticiones repetitivas (*HTTP Polling* cada 1 segundo), se instauró el Controlador v2.

#### Módulo: Simulador Streaming v2 (`/api/v2/simulation`)
**Endpoint Base:** `GET /stream/{sessionId}?limit=220&intervalMs=500`  
**Headers Requeridos (Cliente):** `Accept: text/event-stream`  
**Respuesta:** Flujo infinito (Chunked Transfer) de datos hasta que el servidor declare el fin del túnel.

La arquitectura SSE define la propagación de tres "Tipos de Eventos" por el mismo túnel. React intercepta cada tipo a través de EventListeners.

| Nombre de Evento SSE | Payload Principal (Serializado JSON) | Uso en Cliente (Frontend) |
|---|---|---|
| `snapshot` | `SimulationMapSnapshotDTO` (Estado general, lista de Rutas Activas con campos `from`, `to`, `progress`) | Movilizar las animaciones del mapa interactivo geolocalizado. |
| `kpi` | `SimulationKpiSnapshotDTO` (SLA%, Nodos Críticos, `airportLoads` [diccionario de saturaciones]) | Refrescar indicadores financieros de logística y colorear en rojo/amarillo/verde los aeropuertos saturados. |
| `eventLog` | Cadena (String) con una decisión técnica del backend (Ej: "Ruta redireccionada a Lima por colapso") | Poblar la tabla bitácora visual y notificar alertas temporales. |

### 2.3 Flujo Cronológico: Del Clic al Fin de Simulación

1. **[Solicitud]** Usuario presiona "Ejecutar Simulación".
2. **[Disparador API]** `useControlTowerController` llama mediante POST a `/api/v1/simulation/run/5`.
3. **[Handshake Backend]** Spring Boot genera el `sessionId` "ABC-123", deposita el ID en su concurrencia de memoria, orquesta al algoritmo en un Thread paralelo, y responde a React: `{ "sessionId": "ABC-123" }`.
4. **[Apertura de Túnel]** React abre un túnel inyectando el ID: `new EventSource('/api/v2/simulation/stream/ABC-123')`.
5. **[Transmisión Constante]** El backend bombea la trama `snapshot` y `kpi` cada medio segundo de vida real.
6. **[Dibujado Fluido]** El UI parsea estas coordenadas parciales moviendo vectores y actualizando KPIs de manera imperceptible.
7. **[Término]** El backend evalúa la metaheurística del último maletín, y envía un evento final con atributo `"status": "DONE"`.
8. **[Clausura]** React intercepta `"DONE"`, ejecuta `EventSource.close()`, deteniendo el streaming. Dispara entonces un GET a `/api/v1/simulation/status/ABC-123` para recolectar el archivo definitivo completo o habilitar los botones de "Exportar Resultados (Excel)".

## 3. Manejo de Errores e Integridad
| Escenario de Falla | Comportamiento Backend | Resolución Frontend |
|---|---|---|
| Sesión No Encontrada | Controlador devuelve 404 (Not Found). | El EventSource intentará reconectar, o React purgará la interfaz si se captura vía polling estático. |
| Colapso Algorítmico Fatal | Pasa a estado `FAILED` en el objeto en memoria, y emite atributo `errorMessage`. | React cierra el SSE, notifica `errorMessage` y retorna a estado **idle de UI**. (Nota: en simulación AS-IS el backend no modela `IDLE`; el idle es del frontend). |
| Pérdida de Conexión Nativa (Red Caída) | El túnel SSE colapsa del lado TCP. | Por estándar, el navegador re-intentará conectar al SSE de forma intrínseca sin necesidad de lógica extra de React. |

## 3.1 Mejoras (sin cambiar contrato)

1. Normalizar y documentar un `status` TO-BE unificado (`IDLE/RUNNING/DONE/FAILED`) para UI, manteniendo el AS-IS del simulador (`RUNNING/DONE/FAILED`).
1. Definir límites y validaciones consistentes para `limit` e `intervalMs` en el endpoint SSE (ya existe sanitización en backend v2).
