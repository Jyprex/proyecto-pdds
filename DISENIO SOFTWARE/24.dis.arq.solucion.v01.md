## 2. Arquitectura de Solución
El sistema Tasf.B2B está construido sobre una arquitectura **Cliente-Servidor de tres capas** altamente asíncrona. La capa de cliente (Frontend React SPA) se comunica con el servidor de aplicaciones (Spring Boot) a través de APIs REST para la persistencia transaccional, y utiliza canales unidireccionales persistentes **Server-Sent Events (SSE)** para la renderización de la simulación continua.

## 2.0 Diseño Objetivo (TO-BE)

La arquitectura objetivo se define como un N-tier pragmático, manteniendo el planificador en proceso (misma JVM) y formalizando el streaming SSE para telemetría.

### 2.1 Capas del Sistema Estructural
| Capa | Componente Principal | Tecnología | Responsabilidad Principal |
|---|---|---|---|
| **Presentación** | Aplicación Single-Page (SPA) | React + Vite + Vanilla CSS | Gestión de UI interactiva (Control Tower), tableros de simulación, y representación SVG de vuelos. |
| **Aplicación** | API Gateway & Business Logic | Spring Boot 3.x + Spring MVC | Controladores REST HTTP, autorización (de existir), endpoints de control de flujo y generación de eventos de simulación (SSE). |
| **Planificación** | Módulo Core Planificador | Java 17+ (Módulo Spring Interno) | Alberga los motores heurísticos y metaheurísticos: **Algoritmo Genético Híbrido (HGA)** y **Adaptive Large Neighborhood Search (ALNS)**. Posee el reloj de simulación asíncrono. |
| **Datos** | Motor de Base de Datos | SQL + Spring Data JPA | Persistencia mínima necesaria: Aeropuertos, Vuelos y Envíos (demanda). |

### 2.2 Arquitectura N-Tier del Backend (Implementación Real)
El backend utiliza los estándares de `DOCUMENTACION/63.std.programacion-backend.v01.md`:
1. **Controllers (`@RestController`):** Reciben tráfico HTTP. `SimulationV2Controller` expone flujos Reactivos usando `SseEmitter` para emitir mapas y KPIs asíncronamente sin bloquear hilos. Los controladores carecen absolutamente de lógica matemática o algorítmica.
2. **Services (`@Service`):** Contienen las reglas de negocio puras. `SimulationService` orquesta la ejecución de los motores algoritmos (HGA/ALNS). Gestiona las transacciones (`@Transactional`) hacia la base de datos de manera aislada.
3. **Session Holders:** Para manejar las simulaciones de alto rendimiento sin sobrecargar la Base de Datos, se emplea un manejador de estado en memoria concurrente (`SimulationProgressHolder`). Este componente almacena transitoriamente el `SimulationSessionState`, funcionando como caché primaria de lectura para los clientes SSE.
4. **Repositories (`JpaRepository`):** Capa final conectada al motor relacional, encargada de la persistencia histórica o la extracción masiva al término de la simulación.

### 2.3 Componente Planificador (Arquitectura en Memoria)
Para la ejecución del "Día a Día" o el modelo de "Colapso", el planificador opera en la misma JVM del backend, compartiendo contexto:
- **Motor HGA:** Utilizado primordialmente en *warm-starts* o al inicio del día. Consolida paquetes en *Super-Lotes* y evoluciona rutas estáticas para minimizar trasbordos.
- **Motor ALNS:** Metaheurística empleada en el replanteo dinámico (*Ruin/Repair*). Cuando hay una caída o congestión crítica en un aeropuerto durante la simulación, el ALNS toma la solución del HGA y re-asigna vuelos iterativamente optimizando el remanente.

### 2.4 Patrones de Comunicación y Flujo
La fluidez de la interfaz requiere esquemas de red que superen el *HTTP Polling* ineficiente:
| Origen | Destino | Protocolo | Propósito Implementado |
|---|---|---|---|
| **Frontend React** | **Spring Boot (v1)** | `HTTP POST/GET` | Invocación de Comandos: "Iniciar Simulación", "Exportar Excel", "Obtener Resumen Final". |
| **Spring Boot (v2)** | **Frontend React** | `HTTP SSE (text/event-stream)` | **Streaming Continuo:** El servidor empuja `snapshots` espaciales (rutas activas), `kpi` (saturación de aeropuertos, `airportLoads`) y `eventLogs` (bitácora textual) a la UI cada 500ms. |
| **Servicios Spring** | **Planificador Core** | `Local In-Memory Method Call` | Invocación asíncrona (`@Async` o `CompletableFuture`) para no bloquear el Hilo Principal del Controller durante la resolución del grafo. |

### 2.5 Decisiones Arquitectónicas (ADRs)
* **DA-01: Migración a SSE en lugar de Polling:** Dado que el Frontend requería graficar vectores (aviones) cada segundo de simulación temporal, el polling saturaba el EventLoop del navegador. Server-Sent Events permite conexiones persistentes unidireccionales (Server-to-Client), ideales para esta telemetría.
* **DA-02: Planificador en el mismo nodo JVM:** Se optó por embeber el core heurístico en el mismo nodo del API web para evitar penalizaciones severas de latencia (Network I/O) al momento de transportar miles de entidades de rutas. En un futuro, podría modularizarse a un gRPC Worker independiente si se escala horizontalmente.
* **DA-03: Exportación Diferida a Excel:** Los resultados analíticos profundos de las iteraciones de la metaheurística se exportan a `.xlsx` (`SimulationExcelService`) en la fase terminal (`status == DONE`) para mitigar impactos de memoria concurrentes.

## 2.6 Implementación Actual (AS-IS) y Mapeo

1. Base de datos del prototipo: H2 en memoria (`spring.datasource.url=jdbc:h2:mem:testdb`).
1. Simulación asíncrona: `@Async("simulationExecutor")` y `AsyncConfig` define `simulationExecutor`/`streamingExecutor`.
1. Streaming: `SimulationV2Controller` usa `SseEmitter` con eventos `snapshot`, `kpi`, `eventLog`.
1. Estado en memoria: `SimulationProgressHolder` (simulación) y `PlanningSessionHolder` (planificador).
