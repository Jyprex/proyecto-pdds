## 1. Introducción

Este documento describe la arquitectura de software adoptada para el sistema Tasf.B2B y establece los principios y decisiones de diseño que deben guiar el desarrollo de todos sus componentes. La arquitectura responde a los requisitos funcionales del caso: registro de envíos, planificación y replanificación de rutas, monitoreo visual en tiempo real y simulación de tres escenarios operacionales (día a día, periodo y colapso). Estas decisiones arquitectónicas son vinculantes para el equipo de desarrollo durante el ciclo de vida del prototipo.

## 2. Visión General del Sistema

El sistema Tasf.B2B se estructura como una aplicación de dos capas desacopladas que se comunican a través de una API REST. El **frontend** es una Single Page Application (SPA) construida en React, responsable de la visualización del mapa operacional, los paneles de monitoreo y los controles de simulación. El **backend** es un servicio REST desarrollado con Spring Boot, responsable de la lógica de negocio, la persistencia de datos, la ejecución de los algoritmos de planificación y la provisión de métricas operacionales.

```
┌─────────────────────────────────┐
│  Frontend (React + Vite)        │  ← Navegador del operador
│  Componente Visualizador        │
│  Controles de Simulación        │
└────────────┬────────────────────┘
             │ HTTP/REST + WebSocket
┌────────────▼────────────────────┐
│  Backend (Spring Boot)          │  ← Servidor de aplicación
│  API REST                       │
│  Planificador (HGA / ALNS)      │
│  Gestión de Estado Operacional  │
└────────────┬────────────────────┘
             │ JPA
┌────────────▼────────────────────┐
│  Base de Datos Relacional       │  ← Persistencia
└─────────────────────────────────┘
```

## 3. Principios Arquitectónicos

**Separación de responsabilidades:** La UI (componentes presentacionales) está desacoplada de la lógica de negocio (custom hooks en el frontend, servicios en el backend) y de la capa de datos (repositorios JPA). Ninguna capa accede directamente a la capa que no es su inmediata vecina en la jerarquía.

**Principio SOLID aplicado a React:** Cada componente de la interfaz cumple el principio de responsabilidad única. `App.jsx` es el contenedor raíz que solo ensambla la UI; `useControlTowerController` concentra toda la lógica de estado y de negocio del frontend.

**Estrategia de algoritmo intercambiable:** Los algoritmos planificadores (HGA y ALNS) se implementan como estrategias intercambiables mediante la interfaz `PlanificadorStrategy`. El escenario activo determina cuál estrategia se ejecuta, sin modificar el código del controlador.

**Inmutabilidad del estado:** Tanto en el frontend (estado React) como en el backend (objetos DTO como `record`), los datos se tratan como inmutables. Las transformaciones producen nuevas instancias en lugar de modificar las existentes.

## 4. Arquitectura del Frontend

La SPA adopta el patrón **Container-Presenter**. `App.jsx` es el único contenedor: consume el hook `useControlTowerController` y distribuye los datos y callbacks a los componentes hijos como props. Los componentes en `src/components/` son puramente presentacionales y no tienen acceso al estado global.

Los custom hooks en `src/hooks/` encapsulan la lógica de estado, los cálculos derivados y la comunicación con el backend. Esta capa de hooks actúa como el equivalente frontend de la capa de servicios del backend. La estructura de directorios agrupa los componentes por dominio funcional: `controlTower`, `floating`, `kpi` y `scenarios`.

## 5. Arquitectura del Backend

El backend implementa la arquitectura de tres capas N-Tier de forma estricta: Controllers → Services → Repositories. No se permite el acceso directo entre capas no adyacentes. Las entidades JPA nunca salen de la capa de datos; los DTOs son el único contrato de comunicación entre capas.

Los algoritmos metaheurísticos se ubican en un módulo separado `planner/` dentro del backend, con su propio paquete de servicios, modelos de dominio y configuración de parámetros. Esta separación permite evaluar y reemplazar implementaciones algorítmicas de forma independiente al resto del sistema.

## 6. Comunicación en Tiempo Real

El monitoreo de operaciones en tiempo real (escenario de día a día) requiere comunicación bidireccional entre el frontend y el backend. Se adopta WebSocket mediante **STOMP sobre SockJS** para la emisión de eventos de estado de vuelos, actualizaciones de posición de equipajes y cambios de capacidad de almacenes. Los endpoints REST se utilizan para operaciones de registro, consulta histórica y configuración de simulaciones.

## 7. Gestión de Escenarios de Simulación

El sistema gestiona tres escenarios distintos, cada uno con su propio conjunto de parámetros y comportamiento visual:

| Escenario | Identificador | Descripción |
|---|---|---|
| Día a día | `vivo` | Operación en tiempo real con actualizaciones por WebSocket |
| Simulación de periodo | `periodo` | Simulación acelerada de 3, 5 o 7 días operacionales |
| Colapso | `colapso` | Simulación hasta saturación total del sistema |

El escenario activo se gestiona en el hook `useControlTowerController` del frontend y se comunica al backend mediante el parámetro `scenarioType` en las solicitudes de planificación. El backend adapta la estrategia algorítmica y los rangos de parámetros del semáforo según el escenario recibido.

## 8. Seguridad

La API REST del backend es protegida mediante Spring Security. Los endpoints de lectura de estado operacional pueden ser accesibles con autenticación básica para múltiples dispositivos simultáneos (conforme al requisito de que cualquier dispositivo pueda conectarse al visualizador). Los endpoints de escritura (registro de envíos, inicio de simulación, replanificación) requieren rol autorizado. La implementación de seguridad detallada se especifica en el estándar 65.std.api-rest.v01.

## 9. Calidad y Proceso

El desarrollo del sistema sigue las prácticas del perfil básico de la norma **NTP-ISO/IEC 29110-5-1-2 (VSE)**, conforme a los requisitos de calidad del proyecto. Esto implica la documentación de los cambios, la revisión por pares antes de integrar código y la ejecución del flujo de validación (`lint → build → QA manual`) antes de cada entrega.
