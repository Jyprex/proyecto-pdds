## 1. Introducción

Este documento define los estándares de programación para el desarrollo del backend del sistema Tasf.B2B, implementado con Java 17+ y el ecosistema Spring Boot. La capa de backend expone las APIs REST que alimentan el visualizador de operaciones y el componente planificador de rutas de equipajes. Los lineamientos aquí establecidos garantizan que el código sea robusto, seguro, transaccional y consistente con las prácticas modernas de ingeniería de software empresarial.

## 2. Alcance

Aplica a todos los módulos del backend del sistema: el servicio de gestión de equipajes, el servicio planificador de rutas (algoritmos metaheurísticos), el servicio de monitoreo en tiempo real y cualquier módulo de integración que se desarrolle en iteraciones futuras.

## 4. Arquitectura en Capas (N-Tier)

El backend aplica estrictamente el patrón de arquitectura en tres capas. Ninguna capa debe ser omitida ni fusionada con otra bajo ninguna circunstancia.

**Capa Web — Controllers:** Los controllers manejan exclusivamente las peticiones HTTP entrantes. Su responsabilidad se limita a recibir la solicitud, validar la estructura del input mediante anotaciones de Jakarta Validation, invocar el servicio correspondiente y retornar el DTO de respuesta. Un controller jamás contiene lógica de negocio ni accede directamente al repositorio.

**Capa de Negocio — Services:** Aquí reside toda la lógica empresarial del sistema, incluyendo las reglas de planificación de rutas, la validación de capacidad de aeropuertos, la asignación de vuelos y la gestión de replanificaciones por cancelaciones. Las transacciones de base de datos se gestionan en esta capa mediante `@Transactional`.

**Capa de Datos — Repositories:** Son interfaces que extienden `JpaRepository` o `CrudRepository`. Solo interactúan con la base de datos; no contienen lógica de transformación ni reglas de negocio.

## 5. Patrón DTO — Data Transfer Objects

El uso de DTOs es obligatorio. Las entidades JPA (`@Entity`) nunca se exponen directamente en los controllers ni se incluyen en las respuestas HTTP. Para cada operación relevante se definen DTOs de entrada (*Request*) y de salida (*Response*).

Los DTOs inmutables se implementan como `record` de Java 17, lo que elimina la necesidad de getters, setters y constructores manuales. Los DTOs que requieren construcción incremental pueden implementarse como clases con `@Builder` de Lombok. Las anotaciones de validación (`@NotNull`, `@NotBlank`, `@Size`, `@Min`, `@Max`) se ubican en los campos de los DTOs de entrada.

```java
// ✅ DTO de entrada como record con validación
public record RegistrarEnvioRequest(
    @NotBlank String codigoAerolinea,
    @NotNull @Min(1) Integer cantidadMaletas,
    @NotBlank String aeropuertoOrigen,
    @NotBlank String aeropuertoDestino
) {}
```

## 6. Gestión Transaccional y Concurrencia

Dado que el sistema gestiona operaciones logísticas con múltiples escrituras concurrentes (asignación de capacidad de almacenes, actualización de rutas en vuelo y registro de estados de maletas), la gestión transaccional es crítica.

La anotación `@Transactional` se aplica a nivel de método en la capa de **Service**, nunca en los controllers ni en los repositories. Para excepciones verificadas que deben desencadenar rollback explícito, se usa `@Transactional(rollbackFor = Exception.class)`. En operaciones de alta concurrencia, como la asignación de capacidad de almacén en un aeropuerto con múltiples vuelos simultáneos, se aplica bloqueo optimista mediante la anotación `@Version` en la entidad JPA correspondiente para prevenir condiciones de carrera.

## 7. Inyección de Dependencias

La inyección de dependencias se realiza **exclusivamente por constructor**. El uso de `@Autowired` en atributos (field injection) está prohibido, ya que dificulta las pruebas unitarias y viola los principios de diseño limpio. Se utiliza la anotación `@RequiredArgsConstructor` de Lombok para generar el constructor de forma automática a partir de los campos declarados como `final`.

```java
// ✅ Inyección por constructor con Lombok
@Service
@RequiredArgsConstructor
public class PlanificadorService {
    private final RutaRepository rutaRepository;
    private final AeropuertoRepository aeropuertoRepository;
    // ...
}
```

## 8. Manejo de Errores

El manejo de errores se centraliza en una clase global anotada con `@RestControllerAdvice`. Está prohibido retornar `ResponseEntity` con mapas genéricos (`Map<String, String>`) o strings planos como respuesta de error desde los controllers.

Se definen excepciones de negocio específicas para cada caso de fallo: `AeropuertoNotFoundException`, `CapacidadInsuficienteException`, `VueloCanceladoException`, `RutaNoDisponibleException`. Cada excepción tiene su manejador correspondiente en el `GlobalExceptionHandler`, que retorna un objeto `ApiError` estandarizado con los campos: `status` (código HTTP), `message` (descripción legible), `timestamp` (fecha y hora del error) y `path` (endpoint afectado).

## 9. Buenas Prácticas con Lombok y JPA

En entidades JPA se emplean `@Getter`, `@Builder` y, con precaución, `@Setter`. Se evita `@Data` en entidades, ya que su implementación automática de `hashCode` y `equals` provoca problemas graves de rendimiento y comportamiento inesperado en relaciones perezosas de JPA. Los repositorios retornan `Optional<T>` para consultas de un único registro; el manejo del `Optional` se resuelve en la capa de Service mediante `.orElseThrow(() -> new EntidadNotFoundException(...))`.

## 10. Estándares de Naming en Java

Los nombres de paquetes siguen la convención `com.tasfb2b.modulo.capa` en minúsculas. Las clases se nombran en `PascalCase` con el sufijo que indica su rol: `EnvioController`, `PlanificadorService`, `AeropuertoRepository`, `RegistrarEnvioRequest`, `RutaResponse`. Los métodos y variables locales usan `camelCase`. Las constantes de clase usan `UPPER_SNAKE_CASE`. Los endpoints REST usan sustantivos en plural y kebab-case: `/api/v1/envios`, `/api/v1/aeropuertos/{codigo}/capacidad`.

## 11. Consideraciones para los Algoritmos Metaheurísticos

El componente planificador implementa dos algoritmos metaheurísticos en Java: el Algoritmo Genético Híbrido (HGA) y la Búsqueda por Vecindad Grande Adaptativa (ALNS). Ambos deben encapsularse en clases de servicio independientes (`HgaPlanificadorService`, `AlnsPlanificadorService`) que implementen una interfaz común (`PlanificadorStrategy`), permitiendo el intercambio en tiempo de ejecución según el escenario activo. Las métricas de desempeño de cada algoritmo (tiempo de ejecución, calidad de solución, número de iteraciones) deben registrarse para su posterior comparación en el panel de experimentación numérica.
