## 1. Introducción

Este documento establece los estándares de programación para el desarrollo de la capa de presentación del sistema Tasf.B2B, construida sobre React 19 y Vite. Su propósito es garantizar que el código sea legible, mantenible, escalable y consistente entre todos los miembros del equipo de desarrollo. Estos lineamientos son de cumplimiento obligatorio para cualquier contribución al repositorio del prototipo.

## 2. Alcance

Aplica a todos los archivos `.jsx` y `.js` ubicados en el directorio `src/`, incluyendo componentes de presentación, custom hooks, módulos de datos y utilitarios. Los archivos de configuración de herramientas (`vite.config.js`, `eslint.config.js`) quedan excluidos de estas convenciones salvo indicación explícita.

## 3. Filosofía y Principios Fundamentales

El desarrollo frontend del proyecto se guía por tres principios irrenunciables. El primero es **Clean Code**: el código debe ser escrito para ser leído por personas, no solo ejecutado por máquinas. El segundo es el **Principio de Responsabilidad Única (SRP)**: un componente, hook o función debe resolver exactamente un problema. Si un componente gestiona llamadas de red, renderiza lógica compleja y administra estado global al mismo tiempo, debe ser dividido. El tercero es la **Inmutabilidad**: el estado en React nunca se muta directamente; siempre se emplea el operador spread (`...`), o métodos funcionales como `map`, `filter` y `reduce` para derivar nuevos estados.

## 4. Stack Tecnológico

| Tecnología | Versión | Rol |
|---|---|---|
| React | 19 | Librería UI principal |
| Vite | 6+ | Bundler y servidor de desarrollo |
| JavaScript | ES2022+ | Lenguaje base |
| ESLint | Flat Config | Análisis estático |

No se usa TypeScript en este proyecto. La validación defensiva se implementa mediante desestructuración con valores por defecto y el uso de `?.` (optional chaining) y `??` (nullish coalescing) en todos los accesos a propiedades de objetos provenientes de APIs o estados externos.

## 5. Convenciones de Nomenclatura

Los archivos de componentes y sus declaraciones usan `PascalCase` (`AirportDetailPanel.jsx`). Los custom hooks usan `camelCase` con el prefijo obligatorio `use` (`useControlTowerController.js`). Las variables y funciones internas usan `camelCase` con verbos de acción descriptivos (`handleTabChange`, `togglePanel`, `showAirportDetail`). Las constantes de configuración globales usan `UPPER_SNAKE_CASE` (`MAX_BAGGAGE_CAPACITY`, `API_BASE_URL`). Los props que representan manejadores de eventos llevan el prefijo `on` (`onTabChange`, `onClose`, `onAirportSelect`), mientras que las funciones que los implementan usan `handle` (`handleTabChange`, `handleClose`).

## 6. Arquitectura de Componentes — Patrón Container-Presenter

El proyecto adopta el patrón **Container-Presenter** de forma explícita. `App.jsx` actúa como el único contenedor de nivel superior: no contiene lógica de negocio ni de estado; solo compone la UI y delega toda la lógica al hook `useControlTowerController`. Los componentes en `src/components/` son presentacionales puros: reciben props, renderizan UI y emiten eventos hacia arriba mediante callbacks.

Esta separación es estricta. Un componente presentacional no debe contener llamadas a APIs, lógica de transformación de datos compleja ni estado que pueda ser compartido entre componentes hermanos. Si un componente presentacional necesita estado local efímero (como `isOpen` para un dropdown interno), puede usar `useState`, pero no `useEffect` con side effects externos.

## 7. Estructura Interna de Componentes

El código dentro de cada componente funcional debe ordenarse en el siguiente bloque de forma consistente:

1. Hooks de estado (`useState`, `useReducer`).
2. Hooks de contexto (`useContext`).
3. Referencias del DOM (`useRef`).
4. Efectos secundarios (`useEffect`) con función de limpieza cuando aplique.
5. Funciones auxiliares y manejadores de eventos.
6. Sentencia `return` con JSX.

Los custom hooks que superen 15 líneas de lógica o que puedan reutilizarse en más de un componente deben extraerse al directorio `src/hooks/`.

## 8. Early Returns y Legibilidad del JSX

Se exige el uso de retornos tempranos para manejar estados de carga o error antes del flujo principal de renderizado. La anidación profunda de condicionales en el JSX es una violación de estilo. El siguiente patrón es el correcto:

```javascript
// ✅ Correcto
if (isLoading) return <LoadingSpinner />
if (error) return <ErrorBanner message={error.message} />
return <MainView data={data} />
```

Las expresiones ternarias en JSX se permiten solo cuando tienen dos ramas simples. Para lógica de tres o más condiciones, se extrae a una variable o función antes del `return`.

## 9. Organización de Directorios

```
src/
├── assets/          # Imágenes y recursos estáticos
├── components/      # Componentes presentacionales agrupados por dominio
│   ├── controlTower/   # Componentes del área principal del mapa
│   ├── floating/       # Paneles flotantes de información
│   ├── kpi/            # Tira de indicadores y controles de simulación
│   └── scenarios/      # Paneles de configuración por escenario
├── data/            # Datos estáticos y mocks de desarrollo
├── hooks/           # Custom hooks con lógica de negocio y estado
├── App.css          # Estilos globales del sistema
├── App.jsx          # Componente raíz (contenedor)
├── index.css        # Tokens y variables CSS globales
└── main.jsx         # Punto de entrada de la aplicación
```

Cada nuevo dominio funcional que se agregue al sistema debe seguir esta estructura de subdirectorios.

## 10. Gestión de Imports

Los imports se ordenan en tres bloques separados por una línea en blanco: (1) paquetes externos (`react`, `react-dom`); (2) estilos locales (`.css`); (3) módulos y assets locales (componentes, hooks, datos). Los imports sin uso deben eliminarse de inmediato; ESLint está configurado para reportarlo como error (`no-unused-vars`).

## 11. Estilo y Formato de Código

El proyecto usa 2 espacios de indentación, comillas simples y omisión de punto y coma al final de sentencias. Las comas finales en objetos y arrays multilínea se mantienen donde ya existe ese patrón en el código adyacente. El linter (`npm run lint`) debe ejecutarse sin errores antes de todo commit. El comando `npm run lint -- --fix` resuelve los errores de formato automáticamente.

## 12. Manejo de Errores en el Frontend

Los errores de operaciones asíncronas deben capturarse explícitamente; no se permite dejar promesas rechazadas sin manejo. La interfaz debe ofrecer siempre un texto de fallback accionable al usuario cuando ocurre un fallo visible. Los mensajes de error deben ser específicos para facilitar el diagnóstico, pero no deben exponer información sensible del sistema (rutas internas, stack traces).

## 13. Validación Previa al Commit

Antes de integrar cualquier cambio al repositorio, el desarrollador debe ejecutar en orden: `npm run lint`, seguido de `npm run build`. La validación de comportamiento funcional se realiza manualmente con `npm run dev`. No existe suite de pruebas automatizadas configurada en esta versión del prototipo.
