## 1. Introducción

El presente documento establece los lineamientos y criterios que deben regir el diseño y la implementación de la interfaz gráfica del sistema Tasf.B2B, una plataforma de monitoreo y gestión logística de equipajes en tiempo real. Los estándares aquí definidos buscan garantizar coherencia visual, usabilidad operacional y accesibilidad en todos los componentes interactivos del sistema, considerando que los usuarios finales son operadores de aeropuerto que trabajan bajo presión y en entornos de alta demanda.

## 2. Alcance

Este estándar aplica a todos los componentes de la capa de presentación del sistema, incluyendo el visualizador de mapa mundial, los paneles flotantes de monitoreo, los controles de simulación, los paneles de detalle de aeropuerto, las tiras de KPIs y los controles de navegación por escenario.

## 3. Principios de Diseño

El diseño visual del sistema debe transmitir seriedad operacional y legibilidad inmediata. Se adopta un paradigma de **torre de control oscura** (*dark control tower*): fondos oscuros con alto contraste en los datos críticos, uso racional del color como indicador semántico y tipografía técnica monospaciada para valores numéricos en tiempo real.

Todo elemento visual tiene una función; la decoración sin propósito informativo está prohibida. La densidad de información debe equilibrarse con la claridad perceptual: los operadores deben poder leer el estado del sistema en menos de tres segundos desde cualquier vista.

## 4. Paleta de Colores

El sistema utiliza un esquema de color funcional basado en variables CSS globales definidas en `src/index.css`. Los colores no pueden definirse como valores hexadecimales en línea dentro de los componentes; deben referenciarse siempre desde el sistema de tokens.

| Token | Uso |
|---|---|
| `--ct-bg-primary` | Fondo principal de la aplicación |
| `--ct-accent` | Acción primaria, elementos activos |
| `--ct-text-primary` | Texto principal de alta legibilidad |
| `--ct-text-muted` | Etiquetas secundarias y metadatos |
| `--ct-border` | Bordes de paneles y separadores |

### 4.1 Sistema de Semáforo Operacional

El proyecto exige el uso de colores de semáforo para indicar el estado de capacidad y operaciones. Los rangos son configurables como parámetros del sistema, pero los colores asociados son fijos:

- **Verde** (`--ct-status-ok`): Operación dentro de parámetros normales. Ocupación de almacén por debajo del umbral de advertencia.
- **Ámbar** (`--ct-status-warn`): Estado de advertencia. Capacidad bajo presión; requiere atención del operador.
- **Rojo** (`--ct-status-critical`): Estado crítico o de colapso. Almacén saturado, vuelo cancelado o ruta bloqueada.

Ningún componente debe aplicar directamente los valores `#22c55e`, `#f59e0b` o `#ef4444`; deben usarse los tokens correspondientes para garantizar coherencia en escenarios de colapso donde el tema visual de la interfaz cambia.

## 5. Tipografía

Se utiliza la familia tipográfica **Inter** para texto de interfaz general y **Roboto Mono** para valores numéricos, coordenadas, métricas operacionales y relojes del sistema. Ambas fuentes se cargan desde Google Fonts.

Los tamaños tipográficos se definen exclusivamente mediante las variables `--ct-font-sm`, `--ct-font-md` y `--ct-font-lg`. No se permite el uso de tamaños absolutos en píxeles fuera del sistema de tokens.

## 6. Estructura de Paneles Flotantes

Los paneles de información del sistema (telemetría, inventario en tránsito, comparación de algoritmos, detalle de aeropuerto, etc.) siguen un patrón visual unificado:

- Cabecera con título en mayúsculas, icono identificador y botón de cierre (`×`).
- Cuerpo de contenido con padding uniforme definido por `--ct-panel-padding`.
- Borde semitransparente con `backdrop-filter: blur()` para mantener la legibilidad sobre el mapa.
- Todos los paneles flotantes deben tener `position: absolute` dentro del área del mapa y respetar las zonas de anclaje definidas: `ct-floating-rail--left` y `ct-floating-rail--right`.

Un panel no puede superponerse con otro panel del mismo rail. El sistema de visibilidad (`panelVisibility`) controla cuáles paneles se muestran; el desarrollador no debe gestionar posicionamiento manual.

## 7. Indicadores de Estado y KPIs

La tira de KPIs (`KpiStrip`) muestra métricas operacionales clave. Cada tarjeta KPI debe incluir: etiqueta descriptiva, valor actual, unidad de medida y, si corresponde, indicador de tendencia (positivo/negativo). Los valores deben actualizarse visualmente con una transición suave (`transition: color 0.3s ease`) para evitar parpadeo abrupto.

El estado de colapso del escenario (`isCollapseScenario`) modifica el comportamiento visual de toda la interfaz: el encabezado adopta el color crítico, el footer muestra el badge de advertencia y los nodos de aeropuerto se renderizan en color rojo. Esta transición de estado se gestiona exclusivamente desde el hook `useControlTowerController`.

## 8. Accesibilidad

Todos los elementos interactivos deben disponer de atributos `aria-label` descriptivos. Las secciones del mapa operacional utilizan `aria-label="Mapa de operaciones"` y los elementos decorativos llevan `aria-hidden="true"`. Los botones sin texto visible deben tener siempre `title` y `aria-label` para lectores de pantalla.

Los contrastes de color entre texto y fondo deben cumplir con un ratio mínimo de **4.5:1** para texto normal y **3:1** para texto de tamaño grande, conforme a las pautas WCAG 2.1 nivel AA.

## 9. Responsividad

El sistema está diseñado para monitores de operaciones de al menos 1366 × 768 píxeles. Los breakpoints definidos en `index.css` no deben modificarse sin validación con el equipo de diseño. El mapa operacional (`ct-map-area`) utiliza dimensiones proporcionales para adaptarse al ancho de la ventana, pero la distribución de paneles se optimiza para resoluciones ≥ 1920 × 1080.

## 10. Convenciones de Nomenclatura CSS

Todas las clases CSS del sistema siguen el prefijo `ct-` (Control Tower) para evitar colisiones con librerías de terceros. Los modificadores de estado usan el patrón BEM adaptado: `ct-elemento--modificador`. Por ejemplo: `ct-footer--collapse`, `ct-map-overlay--collapse`, `control-tower--collapse`.

No se permite el uso de clases utilitarias de librerías externas (TailwindCSS, Bootstrap) en este proyecto. Todo estilo se gestiona a través de las hojas de estilo del componente o desde `App.css` e `index.css`.
