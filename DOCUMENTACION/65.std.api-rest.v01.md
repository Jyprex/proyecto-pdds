## 1. Introducción

Este documento establece los estándares para el diseño, versionamiento, seguridad y documentación de la API REST que provee el backend del sistema Tasf.B2B al frontend visualizador. Una API bien diseñada reduce la fricción de integración entre capas, facilita el mantenimiento evolutivo y garantiza contratos claros y estables entre el equipo de backend y el de frontend. Estos lineamientos son de cumplimiento obligatorio para todos los endpoints que se expongan.

## 2. Alcance

Aplica a todos los endpoints HTTP expuestos por el backend Spring Boot, incluyendo los servicios de gestión de envíos, planificación de rutas, consulta de aeropuertos, estado de vuelos y métricas de simulación. Los endpoints de WebSocket para comunicación en tiempo real se documentan de forma complementaria pero no son el foco central de este estándar.

## 3. Estructura y Nomenclatura de Endpoints

Todos los endpoints siguen el estilo RESTful con las siguientes convenciones. Los recursos se nombran con **sustantivos en plural** y en **kebab-case**: `/envios`, `/aeropuertos`, `/vuelos`, `/rutas-planificadas`. Los identificadores de recurso se expresan como segmentos de ruta: `/aeropuertos/{codigo}`, `/envios/{id}/maletas`. Las acciones que no se mapean naturalmente como operaciones CRUD se expresan como sub-recursos o usando verbos en acciones específicas: `/envios/{id}/replanificar`.

La versión de la API se incluye como primer segmento de ruta: `/api/v1/`. Todos los endpoints actuales pertenecen a la versión 1. Un cambio que rompa la compatibilidad con el frontend existente requiere incrementar la versión a `/api/v2/`.

### 3.1 Mapeo de Verbos HTTP

| Verbo | Uso |
|---|---|
| `GET` | Consulta de recursos, sin efecto en el estado del servidor |
| `POST` | Creación de nuevos recursos o inicio de procesos |
| `PUT` | Actualización completa de un recurso existente |
| `PATCH` | Actualización parcial de un recurso existente |
| `DELETE` | Eliminación de un recurso |

## 4. Estructura de Respuestas

### 4.1 Respuestas Exitosas

Las respuestas exitosas retornan el recurso o colección directamente como cuerpo JSON. El código HTTP debe reflejar con precisión el resultado de la operación: `200 OK` para consultas y actualizaciones, `201 Created` para creación de recursos (con el header `Location` apuntando al nuevo recurso), `204 No Content` para eliminaciones exitosas sin cuerpo de respuesta.

```json
// GET /api/v1/aeropuertos/LIM — 200 OK
{
  "codigo": "LIM",
  "nombre": "Aeropuerto Internacional Jorge Chávez",
  "ciudad": "Lima",
  "continente": "America",
  "capacidadAlmacen": 650,
  "ocupacionActual": 312
}
```

### 4.2 Respuestas de Error

Todos los errores retornan un objeto `ApiError` estandarizado. Está prohibido retornar strings planos, HTML o mapas genéricos como cuerpos de error. El objeto de error tiene la siguiente estructura:

```json
{
  "status": 404,
  "error": "Not Found",
  "message": "No se encontró el aeropuerto con código 'XYZ'.",
  "path": "/api/v1/aeropuertos/XYZ",
  "timestamp": "2026-04-21T22:35:00Z"
}
```

### 4.3 Códigos de Estado HTTP

| Código | Significado | Cuándo usarlo |
|---|---|---|
| `200` | OK | Consulta o actualización exitosa |
| `201` | Created | Recurso creado exitosamente |
| `204` | No Content | Eliminación o acción sin datos de retorno |
| `400` | Bad Request | Input inválido, validación fallida |
| `401` | Unauthorized | Falta de autenticación |
| `403` | Forbidden | Sin permiso para el recurso |
| `404` | Not Found | Recurso inexistente |
| `409` | Conflict | Estado inconsistente (ej. capacidad excedida) |
| `422` | Unprocessable Entity | Entidad semánticamente inválida |
| `500` | Internal Server Error | Error no controlado del servidor |

## 5. Endpoints Principales del Sistema

### 5.1 Gestión de Envíos

| Método | Endpoint | Descripción |
|---|---|---|
| `POST` | `/api/v1/envios` | Registra un nuevo envío de maletas |
| `GET` | `/api/v1/envios/{id}` | Consulta el detalle de un envío |
| `GET` | `/api/v1/envios/{id}/plan-viaje` | Obtiene el plan de viaje de las maletas |
| `POST` | `/api/v1/envios/{id}/replanificar` | Dispara la replanificación de ruta |

### 5.2 Aeropuertos y Vuelos

| Método | Endpoint | Descripción |
|---|---|---|
| `GET` | `/api/v1/aeropuertos` | Lista todos los aeropuertos operativos |
| `GET` | `/api/v1/aeropuertos/{codigo}` | Detalle y métricas de un aeropuerto |
| `GET` | `/api/v1/vuelos` | Lista vuelos activos en el sistema |
| `GET` | `/api/v1/vuelos/{id}/estado` | Estado actual de un vuelo |

### 5.3 Planificación y Simulación

| Método | Endpoint | Descripción |
|---|---|---|
| `POST` | `/api/v1/simulacion/iniciar` | Inicia una simulación con los parámetros dados |
| `POST` | `/api/v1/simulacion/pausar` | Pausa la simulación en curso |
| `POST` | `/api/v1/simulacion/detener` | Detiene y reinicia el estado de simulación |
| `GET` | `/api/v1/simulacion/metricas` | Obtiene métricas de desempeño en tiempo real |

## 6. Paginación y Filtrado

Los endpoints que retornan colecciones implementan paginación mediante los parámetros de query: `page` (número de página, base 0), `size` (elementos por página, máximo 100) y `sort` (campo y dirección, ej. `sort=fechaCreacion,desc`). La respuesta paginada sigue la estructura estándar de Spring Data:

```json
{
  "content": [...],
  "totalElements": 1450,
  "totalPages": 15,
  "number": 0,
  "size": 100
}
```

El filtrado se implementa mediante parámetros de query descriptivos: `/api/v1/vuelos?origen=LIM&destino=MAD&estado=en-vuelo`.

## 7. Seguridad de la API

La autenticación se implementa mediante **JWT (JSON Web Tokens)**. Cada solicitud a endpoints protegidos debe incluir el token en el header `Authorization: Bearer <token>`. Los tokens tienen una vigencia de 8 horas para sesiones de operadores. El backend valida el token en cada solicitud mediante un filtro de Spring Security.

Los endpoints de consulta del estado operacional (aeropuertos, vuelos, métricas del mapa) pueden accederse con el rol `ROLE_VIEWER`, asignado a cualquier dispositivo conectado al sistema. Los endpoints de escritura (registro de envíos, inicio de simulación, replanificación) requieren el rol `ROLE_OPERATOR`. La configuración CORS del backend autoriza el origen del frontend en desarrollo (`http://localhost:5173`) y el dominio de producción definido en las variables de entorno.

## 8. Headers Obligatorios

Todas las solicitudes con cuerpo JSON deben incluir el header `Content-Type: application/json`. Las respuestas del servidor incluyen siempre `Content-Type: application/json` y el header `X-Request-Id` con un UUID único por solicitud, útil para el trazado de errores en logs.

## 9. Documentación de la API

La API se documenta automáticamente mediante **SpringDoc OpenAPI** (Swagger UI), accesible en `/api/swagger-ui.html` durante el entorno de desarrollo. Cada endpoint debe incluir las anotaciones `@Operation`, `@ApiResponse` y `@Parameter` para garantizar que la documentación generada sea completa y útil para el equipo de frontend.
